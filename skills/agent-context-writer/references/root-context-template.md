# Root Context Template

根目录上下文只负责组织全局内容，不负责决定工具名、文件名或放置位置。那些规则在主文档和工具 reference 里。

根目录文件只写整个项目都成立的事实。不要把某个单独模块的实现细节塞进来。

通常会覆盖这些内容：

1. 项目定位
2. 顶层目录地图
3. 全局开发命令
4. 全局验证方式
5. 仓库级约定
6. 高风险区域和禁区

可以按下面的骨架组织，但不要机械保留空标题。

```md
# <项目名或工作区名>

## 项目定位
- 这个仓库主要做什么
- 主要语言、框架、运行方式
- 哪些目录是主入口

## 目录地图
- `apps/...`: ...
- `packages/...`: ...
- `src/...`: ...
- `scripts/...`: ...

## 常用命令
- 安装依赖: `...`
- 本地开发: `...`
- 测试: `...`
- lint / format / typecheck: `...`
- 构建: `...`

## 全局约定
- 修改代码前应先阅读哪些目录或文件
- 跨目录通用的实现约定
- 哪些生成文件、锁文件、快照文件不能随意改

## 验证要求
- 做完改动后最少需要跑哪些检查
- 哪些检查代价高，只在必要时执行

## 风险提示
- 哪些目录或脚本容易造成破坏性影响
- 哪些基础设施、迁移、部署相关改动要特别谨慎
```

写作时注意：

- 命令必须来自仓库事实，不要脑补
- 目录地图只列真正重要的顶层结构
- 如果仓库是 monorepo，要说明哪些目录是 app、哪些是 package、哪些是共享基础设施
- 如果多个子目录会有自己的上下文文件，根目录只保留全局规则，不要提前写太多局部内容
